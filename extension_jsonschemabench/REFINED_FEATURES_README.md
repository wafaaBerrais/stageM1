# Refined Feature Dictionary And Modeling Recommendation

This README documents the refined features generated by:

- `extension_jsonschemabench/scripts/analyze_refined_features.py`
- `extension_jsonschemabench/scripts/analyze_refined_features_v2.py`

The current global analyses are stored under:

- `guidance/Github_global/refined_feature_analysis_v2`
- `xgr/Github_global/refined_feature_analysis_v2`
- `outlines/Github_global_no_hard/refined_feature_analysis_v2`

`outlines/Github_hard` is excluded for now.

## Goal

The goal is not to keep every refined feature for modeling.

The goal is to keep the smallest feature set that captures the largest UNDER/OVER risk signal. The refined feature tables are intentionally richer than the final modeling feature set so that we can inspect causes, compare frameworks, then prune aggressively.

Recommended modeling rule:

- Keep a refined feature if it has enough support and clear lift.
- Use `under_invalid_only_risk.csv` for UNDER, because UNDER only makes sense among invalid tests.
- Use `over_valid_only_risk.csv` for OVER, because OVER only makes sense among valid tests.
- Ignore UNDER lift for `guidance` global for now, because global guidance has baseline UNDER rate 0.

Suggested default thresholds:

- `low_support=false`
- `support_schemas >= 10`
- `support_invalid_tests >= 20` for UNDER, or `support_valid_tests >= 20` for OVER
- lift `>= 1.5` in at least one framework, or lift `>= 1.25` in at least two frameworks
- prefer features with an interpretable validation mechanism, not only broad presence flags

For a compact model, I recommend starting with the `Core Modeling Set` below and treating the rest as exploratory.

## Core Modeling Set

These are the features I would keep first because they repeatedly represent concrete failure mechanisms.

### Numeric UNDER

Keep:

- `numeric_boundary_case`
- `numeric_target_type`
- `numeric_parent_keyword`
- `numeric_property_required`
- `numeric_is_in_properties`
- `numeric_has_default`
- `numeric_has_min_and_max`

Meaning:

- `numeric_boundary_case`: whether the instance is below minimum, above maximum, violates multipleOf, equals a boundary, or is inside range.
- `numeric_target_type`: numeric schema target, such as integer, number, mixed, or absent.
- `numeric_parent_keyword`: where the numeric constraint appears, for example properties, items, oneOf, allOf.
- `numeric_property_required`: whether the numeric property is required.
- `numeric_is_in_properties`: whether the numeric constraint is under object properties.
- `numeric_has_default`: whether the numeric schema node also has a default.
- `numeric_has_min_and_max`: whether lower and upper numeric bounds both exist.

Why keep:

- This is the strongest UNDER family for `outlines`.
- It is also important for `xgr`.
- The features are mechanistic, not just broad keywords.

### Object/Properties UNDER

Keep selectively:

- `object_required_case`
- `object_missing_required_count_bucket`
- `object_property_count_boundary_case`
- `object_additional_properties_case`
- `object_extra_properties_count_bucket`
- `object_additionalProperties_value`
- `object_has_minProperties`
- `object_has_maxProperties`

Meaning:

- `object_required_case`: whether required properties are missing or all required properties are present.
- `object_missing_required_count_bucket`: bucketed number of missing required fields.
- `object_property_count_boundary_case`: whether the object is below minProperties, above maxProperties, or inside bounds.
- `object_additional_properties_case`: whether the instance has extra properties and whether they are disallowed, allowed, or validated by a schema.
- `object_extra_properties_count_bucket`: bucketed number of unmatched/extra properties.
- `object_additionalProperties_value`: schema-level value: absent, true, false, schema, or mixed.
- `object_has_minProperties`, `object_has_maxProperties`: whether object count bounds exist.

Why keep selectively:

- This family is useful for UNDER because object constraints can be accepted too permissively.
- It is less dominant than numeric for `outlines`, but it is meaningful for `xgr`.
- Do not keep all object count/context columns if lift is weak.

### Combinator UNDER And OVER

Keep:

- `combinator_type`
- `combinator_branch_count_bucket`
- `allOf_satisfied_branch_count_bucket`
- `allOf_satisfied_branch_ratio_bucket`
- `anyOf_satisfied_branch_count_bucket`
- `oneOf_satisfied_branch_count_bucket`
- `branches_have_required`
- `branches_have_properties`
- `branches_have_not`
- `branches_have_numeric`
- `branches_have_object_bounds`
- `branches_overlapping_properties`
- `same_node_allOf_and_required`
- `same_node_allOf_and_properties`
- `same_node_oneOf_and_required`
- `same_node_combinator_and_numeric`
- `same_node_combinator_and_properties`

Meaning:

- `combinator_type`: absent, allOf, anyOf, oneOf, or mixed.
- `combinator_branch_count_bucket`: bucketed max number of branches.
- `*_satisfied_branch_count_bucket`: how many branches are satisfied by the test instance.
- `allOf_satisfied_branch_ratio_bucket`: whether no, partial, all, or more-than-all branches are satisfied.
- `branches_have_*`: whether combinator branches contain specific risky keywords.
- `branches_overlapping_properties`: whether branches define overlapping object properties.
- `same_node_*`: whether risky keywords occur together at the same schema node.

Why keep:

- For `xgr`, combinator UNDER signals are strong, especially partial/allOf and oneOf multi-match cases.
- For OVER, patternProperties and combinator interactions are also strong.
- These features explain branch interaction, which a simple `has_allOf` or `has_oneOf` cannot capture.

### Not UNDER/OVER

Keep selectively:

- `instance_satisfies_not_subschema`
- `not_parent_keyword`
- `not_target_type`
- `not_contains_required`
- `not_contains_properties`
- `not_contains_numeric`
- `not_contains_enum`
- `same_node_not_and_properties`
- `same_node_not_and_required`
- `same_node_not_and_enum`

Meaning:

- `instance_satisfies_not_subschema`: whether the instance satisfies the forbidden `not` subschema.
- `not_parent_keyword`: where `not` appears, such as properties, allOf, oneOf, root.
- `not_target_type`: target type inside the `not` subschema.
- `not_contains_*`: whether the `not` subschema contains specific constraints.
- `same_node_not_and_*`: whether `not` appears on the same node as other risky constraints.

Why keep selectively:

- `not` can be dangerous, but support is sometimes small.
- Keep these when lift and support are sufficient, especially for `xgr`.

### PatternProperties OVER

Keep:

- `additionalProperties_value`
- `patternProperties_with_properties`
- `patternProperties_has_additionalProperties`
- `patternProperties_regex_has_anchor`
- `patternProperties_regex_has_dotstar`
- `patternProperties_regex_has_alternation`
- `patternProperties_regex_has_charclass`
- `instance_has_unmatched_keys`
- `instance_matching_pattern_keys_count_bucket`

Meaning:

- `additionalProperties_value`: absent, true, false, schema, mixed.
- `patternProperties_with_properties`: whether named properties and patternProperties coexist.
- `patternProperties_has_additionalProperties`: whether additionalProperties is explicitly present.
- regex flags: whether patternProperties regexes contain anchors, dot-star, alternation, or character classes.
- `instance_has_unmatched_keys`: whether the instance has keys matching neither named nor pattern properties.
- `instance_matching_pattern_keys_count_bucket`: how many instance keys match patternProperties.

Why keep:

- This is one of the strongest OVER families across frameworks.
- It directly explains over-rejection of valid objects.

## Exploratory Or Optional Features

These were generated for completeness and diagnosis, but I do not recommend keeping all of them in the first modeling feature set.

### Enum/Const

Generated:

- `enum_validation_case`
- `enum_target_type`
- `enum_parent_keyword`
- `enum_size_bucket`
- `enum_contains_null`
- `enum_contains_mixed_types`
- `const_validation_case`
- `const_target_type`
- `const_parent_keyword`
- `const_type`

Meaning:

- Tracks enum/const mismatch, target type, parent location, enum size, and mixed enum values.

Recommendation:

- Keep only if lift passes threshold.
- Useful as a small optional family because it is interpretable.
- Do not keep all enum metadata by default.

### String

Generated:

- `string_validation_case`
- `string_target_type`
- `string_parent_keyword`
- `string_has_pattern`
- `string_has_minLength`
- `string_has_maxLength`
- `string_has_min_and_max_length`
- `string_has_format`
- `string_format_value`

Meaning:

- Tracks pattern violations, length bound violations, and format-bearing string schemas.

Recommendation:

- Do not keep the full string family by default.
- Keep only `string_validation_case` and maybe `string_format_value` if they pass lift/support.
- In the current studies, string is not the main global risk driver compared with numeric, object, not, combinators, and patternProperties.

### Array/Items

Generated:

- `array_validation_case`
- `array_invalid_items_count_bucket`
- `array_target_type`
- `array_parent_keyword`
- `array_items_kind`
- `array_prefixItems_count_bucket`
- `array_has_minItems`
- `array_has_maxItems`
- `array_has_min_and_max_items`
- `array_has_uniqueItems`
- `array_has_contains`
- `array_has_minContains`
- `array_has_maxContains`

Meaning:

- Tracks item validation, minItems/maxItems, contains, uniqueItems, and prefixItems contexts.

Recommendation:

- Do not keep the full array family by default.
- Keep `array_validation_case`, `array_has_minItems`, `array_has_maxItems`, and `array_has_min_and_max_items` only if they remain high lift after comparing frameworks.
- For `xgr`, array bounds showed a meaningful UNDER signal; for `outlines`, it is weaker.

## Practical Pruning Plan

For the next modeling dataset, I recommend three tiers.

### Tier 1: Compact Core

Use this first:

- Numeric UNDER core
- Object/property selected features
- Combinator branch/satisfaction features
- Not selected features
- PatternProperties OVER core

This is the "minimum maximum-risk" feature set.

### Tier 2: Add If Needed

Add only if cross-framework lift/support justifies it:

- `enum_validation_case`
- `enum_size_bucket`
- `const_validation_case`
- `array_validation_case`
- `array_has_minItems`
- `array_has_maxItems`
- `string_validation_case`
- `string_format_value`

### Tier 3: Exploration Only

Keep out of the first model unless ablation says they help:

- Most broad counts
- Most parent/type metadata
- Most string flags
- Most array flags
- Rare low-support local pair flags

## Important Notes

- A high lift is correlational, not proof of cause.
- Low support can create unstable high lift.
- If two features encode almost the same thing, keep the more mechanistic one.
  Example: prefer `numeric_boundary_case=below_min` over `has_minimum=true`.
- For UNDER, compare invalid-only rates.
- For OVER, compare valid-only rates.
- For final training, we should create a pruned feature export rather than using the full `refined_test_features.csv` directly.
